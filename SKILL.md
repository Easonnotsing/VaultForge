# Obsidian Learning Material Processing

将学习资料转化为 Obsidian 结构化原子笔记知识库的完整流程。

## 核心原则

### ⚠️ 重要：必须完整阅读所有指定文件

**绝对禁止基于部分内容生成学习路线图或笔记。**

- 必须读取用户选择的**所有**学习文件
- **优先一次性完整读取整个文件**，仅在遇到技术限制时才分批
- 如果必须分批读取（例如 PDF 每批 50 页），**自动继续无需用户确认**
- 只有在确认已读取全部内容后，才能生成学习路线图

### ⚠️ 重要：原子笔记内容必须丰富详尽

**原子笔记不是提纲，而是要让初学者能够完整理解和学习的教学材料。**

每个原子笔记必须包含：
- **详细解释**：不少于 200 字的核心概念讲解，背景、定义、原理、应用场景
- **案例说明**：来自学习材料的完整案例，附带背景、过程、结果分析
- **原文引用**：标注准确的文档名和页码
- **核心思考**：促进深度理解的思考问题，而非简单提问

如果笔记内容过于简化（如仅有 bullet points），必须返工重写。

### 批量确认原则

**避免逐个确认，采用批量确认模式：**
- Phase 1：路线图生成后，一次性展示完整版，征询"继续"或"修改意见"
- Phase 2：文件结构创建完成后，一次性展示所有创建的文件夹和文件
- Phase 3：内容填充完成后，一次性展示填充结果摘要
- Phase 4：双链建立完成后，一次性展示链接摘要

### 双链建立标准

**双链不只是关键词关联，而是基于逻辑相关性：**

1. **推导关系**：一篇笔记的结论由另一篇推导而出
2. **原理相似**：两篇笔记的原理可以类比借鉴、触类旁通
3. **结论矛盾**：两篇笔记的结论存在矛盾，需要深入辩论对比
4. **应用关联**：一篇笔记的应用场景涉及另一篇的概念
5. **背景关联**：两篇笔记属于同一领域的不同发展阶段或不同方面

**禁止基于纯粹术语相似而建立双链。**

## 使用场景

- 用户提供学习材料（PDF、文档等）
- 需要生成学习路线图
- 需要创建原子笔记和 MOC
- 需要建立笔记间双链

## 完整流程

### Phase 1: 学习路线图生成

**步骤 1.1: 检测 vault 并选择文件夹**
- 自动检测 Obsidian vault 根目录
- 使用网页选择器或文字列表让用户选择文件夹
- 用户确认选择后进入下一步

**步骤 1.2: 完整阅读学习材料**
- 优先**一次性完整读取**每个文件
- 如果文件过大（如 PDF），自动分批读取（每批 50 页），**无需用户确认**
- 实时报告进度：`📖 正在读取：文件名.pdf - [150/329 页] (46%)`
- 完成后报告：`✅ 阅读完成！共 X/Y 页 (100%)`

**步骤 1.3: 生成并确认路线图**
- 基于完整阅读内容生成学习路线图
- 展示**完整版**路线图（含详细知识点、案例、原文引用）
- 用户回复"继续"则进入保存；回复"修改+具体意见"则修改后再确认

**步骤 1.4: 保存两份文件**
- `学习路线图（完整版） - {主题名}.md` - 详细版
- `学习路线图 - {主题名}.md` - 大纲版（仅 H2/H3/bullet）

### Phase 2: 文件结构创建

**步骤 2.1: 批量创建文件夹和文件**
- 按 H2/H3 层级创建文件夹结构
- 在每个 H3 主题文件夹创建 MOC 笔记
- 为每个 bullet 知识点创建原子笔记文件
- 创建完成后**一次性展示**所有创建的文件列表

**步骤 2.2: 填充 MOC 内容**
- 将原子笔记链接写入对应 MOC
- 在路线图大纲中将 H3 与对应 MOC 建立双向链接

**步骤 2.3: 验证结构**
- 检查文件夹结构、笔记数量与路线图是否一致
- 如有不一致，修复后再进入 Phase 3

### Phase 3: 内容生成（并行）

**步骤 3.1: 分配并行任务**
- 原子笔记 ≤ 10：2 个并行 agents
- 原子笔记 11-30：3-4 个并行 agents
- 原子笔记 > 30：5 个并行 agents

**步骤 3.2: 批量填充内容**
- 每个 agent 接收多个笔记的填充任务（3-5 个）
- 所有 agents **并行执行**
- 完成后一次性展示填充结果摘要

**步骤 3.3: 审查质量（可选）**
- 如用户要求，使用 note-reviewer agent 审查
- 否则跳过此步骤

### Phase 4: 双链建立

**步骤 4.1: 读取所有原子笔记**
- 批量读取所有原子笔记内容
- 分析笔记间的逻辑关联

**步骤 4.2: 建立笔记间双链**
- 根据双链标准建立关联
- 在相关笔记的 `## 相关笔记` 部分添加链接

**步骤 4.3: 建立路线图与 MOC 的双链**
- 在大纲版路线图中，每个 H3 主题后添加指向对应 MOC 的链接
- 在每个 MOC 笔记中，添加指向路线图大纲的链接
- 示例：
  ```markdown
  <!-- 路线图大纲中 -->
  ## 02. 客户领域

  [[02. 客户领域/客户战略核心/客户战略核心 MOC|客户战略核心]]

  <!-- 对应 MOC 中 -->
  ## 相关笔记

  - [[../../学习路线图 - Digital Transformation Roadmap|学习路线图]]
  ```

**步骤 4.4: 批量报告结果**
- 展示所有添加的双链摘要
- 确认完成

### Phase 5: 最终审查与核心问题生成

**步骤 5.1: 审查所有原子笔记**

必须逐项检查以下内容，**发现问题必须修复**：

#### 5.1.1 原子笔记完整性检查
- [ ] 每个原子笔记文件都已创建
- [ ] 笔记内容符合丰富度要求（核心概念 200+ 字、案例完整、原文引用准确）
- [ ] 笔记内容无大量重复
- [ ] 笔记内容无明显错误或曲解

#### 5.1.2 MOC 与原子笔记双链检查
- [ ] 每个 MOC 文件中已添加 `## 相关笔记` 部分
- [ ] 该 MOC 文件夹下的所有原子笔记都已用 wikilink 链接到 MOC
- [ ] 链接格式正确：`[[filename|显示文本]]`
- 示例：
  ```markdown
  ## 相关笔记
  
  - [[原子笔记A]]
  - [[原子笔记B]]
  - [[原子笔记C]]
  ```

#### 5.1.3 路线图与 MOC 双链检查
- [ ] 大纲版学习路线图中每个 H3 主题后已添加指向对应 MOC 的链接
- [ ] 每个 MOC 笔记中已添加指向大纲版学习路线的链接
- 示例：
  ```markdown
  <!-- 大纲版路线图中 -->
  ## 02. 客户领域
  
  [[02. 客户领域/客户战略核心/客户战略核心 MOC|客户战略核心]]
  
  <!-- 对应 MOC 中 -->
  ## 相关笔记
  
  - [[../../学习路线图 - Digital Transformation Roadmap|学习路线图]]
  ```

**步骤 5.2: 生成"核心问题"笔记**

在学习材料所在文件夹下生成新笔记 `核心问题.md`：

#### 5.2.1 问题要求

**数量限制**：不超过 5 个核心问题

**问题质量标准**：
1. **精华性**：问题应涵盖学习材料最核心的知识、原则、方法论等精华内容
2. **引导性**：问题应引导用户更高效地带着问题去学习，而非简单的信息检索
3. **逻辑性**：问题之间应有逻辑层次，不应有明显的重复和矛盾

#### 5.2.2 问题结构

每个问题应包含：
- **问题标题**：简明扼要的核心问题
- **问题背景**：解释这个问题的背景、重要性、要解决什么
- **子问题**（可选）：将复杂问题拆分为 2-3 个子问题

#### 5.2.3 输出格式

```markdown
---
title: 核心问题
date: YYYY-MM-DD
tags:
  - 核心问题
  - {主题名}
---

# 核心问题

> 本笔记整理自学习材料的核心精华，通过不超过 5 个关键问题引导高效学习。

**问题背景**：{解释为什么这个问题重要，要解决什么，背景是什么}

{子问题 1.1}
{子问题 1.2}

---

## 问题二：{问题标题}

**问题背景**：{...}

---

## 问题三：{问题标题}

**问题背景**：{...}

---

## 问题四：{问题标题}

**问题背景**：{...}

---

## 问题五：{问题标题}

**问题背景**：{...}
```

#### 5.2.4 问题列表示例

假设学习材料是"平台战略"：
```
1. 平台商业模式的核心要素是什么？如何区分平台与传统管道商业模式？

2. 平台企业如何创造和获取价值？价值创造与价值捕获的区别是什么？

4. 平台扩张的策略选择：渗透策略 vs 扩张策略何时更有效？
   - 子问题：如何识别和突破平台扩张的临界点？

5. 监管环境如何影响平台战略？平台企业面临哪些合规挑战？
   - 子问题：网络效应与市场支配力之间的法律边界在哪里？
```

**步骤 5.3: 批量报告最终结果**

一次性展示：
- 原子笔记总数、创建状态、内容质量审查结果
- MOC 与原子笔记双链建立情况
- 路线图与 MOC 双向链接建立情况
- "核心问题"笔记已生成

如有任何检查项未通过，修复后再报告最终结果。

### Phase 6: 深度研究与争议分析

**步骤 6.1: 执行 Deep Research**

使用 deep-research 技能对学习主题进行深度研究：

#### 6.1.1 研究范围
- 该领域的主要理论框架和基础概念
- 学界/业界的主流观点和新兴观点
- 不同的实践方法和成功案例
- 存在的争议、分歧和未解决的问题

#### 6.1.2 研究要求
- 搜索该领域的权威来源（学术论文、行业报告、专家观点）
- 识别至少 5-10 个有价值的参考来源
- 特别关注与学习材料观点不一致的内容

#### 6.2.2 内容填充指南

**毋庸置疑部分**：
- 寻找多个来源一致认可的观点
- 解释为何这些观点被广泛接受
- 引用权威来源支持

**争议部分**：
- 识别学习材料与其他来源的分歧点
- 客观呈现各方观点，不偏袒任何一方
- 指出争议的核心和根源

**情景依赖部分**：
- 识别方法论层面的建议
- 解释不同情景下的适用性差异
- 找出关键决定变量

**问题清单**：
- 问题应具有开放性和深度
- 避免 yes/no 类问题
- 优先选择 expert 们经常讨论的核心议题

## 失败分析与改进

### 历史失败案例与解决方案

| 失败场景 | 原因分析 | 解决方案 |
|---------|---------|---------|
| Sub-agents 无法访问 vault 文件 | 沙箱权限限制 | **直接使用 Python 脚本**而非调用 sub-agents |
| 逐个确认导致进度缓慢 | 交互模式设计问题 | 改为**批量确认模式** |
| 长文件分批阅读需要多次确认 | 未考虑自动继续 | 改为**自动分批，无需确认** |
| 原子笔记内容过于简化 | 未明确内容丰富度要求 | 明确要求**200+ 字详细解释，含案例分析** |
| 双链标准模糊 | 仅基于术语相似 | 明确**5 种逻辑关联标准** |
| 路线图与 MOC 未建立双链 | 步骤遗漏 | 增加**步骤 4.3 专门建立此双链** |
| MOC 与原子笔记双链遗漏 | 步骤遗漏 | 增加 **Phase 5.1.2 专门检查 MOC 与原子笔记双链** |

### 稳定性保障

1. **优先使用 Python 脚本而非 agents** - agents 受限于权限和上下文
2. **批量操作减少交互** - 减少用户确认次数，提高效率
3. **自动错误恢复** - 脚本出错时记录日志并继续
4. **进度持久化** - 长时间任务定期保存进度

## 输出质量标准

### 路线图质量
- 完整版：详细知识点描述（200+ 字）、完整案例说明、准确原文引用
- 大纲版：严格仅含 H2/H3/bullet
- 必须基于完整阅读内容

### 原子笔记质量
- **内容丰富度**：核心概念讲解 200+ 字，案例完整分析
- **初学者友好**：无基础读者也能理解概念
- **案例支撑**：必须有来自学习材料的案例
- **思考引导**：促进深度理解的反思问题
- 无大量重复内容

### 双链质量
- 笔记间双链基于逻辑关联（非术语相似）
- 路线图与 MOC 双向链接
- MOC 正确链接所有原子笔记

### Obsidian 格式
- frontmatter 包含 title、date、tags
- wikilinks 目标笔记存在
- 格式规范统一

### 核心问题质量
- 不超过 5 个核心问题
- 每个问题包含问题背景（可选子问题）
- 问题涵盖核心知识、原则、方法论
- 问题具有引导性，能促进深度学习
- 问题之间有逻辑层次，无明显重复或矛盾

### 争议分析质量
- 基于深度研究，识别至少 3-5 个有价值的参考来源
- 共识部分有多个来源支持
- 争议部分客观呈现各方观点
- 情景依赖部分说明关键决定变量
- 探讨问题清单具有开放性和深度
- 参考来源按相关性排列，附简短评价
