### 1.2 新认知维度分析

#### 社会性学习动机：成就分享卡片

成就分享卡片（Step 6.5）不只是工程功能。从认知科学角度看，这是将 **Social Recognition（社会认可）** 引入知识建构流程。Bandura 的社会学习理论指出，观察他人成就和自我展示是维持学习动机的关键机制。VaultForge 的卡片将一次学习成果可视化、可分享，满足了两层需求——

1. **自我确认**——看到 "51.5k 字 → 45 篇笔记 → 18 条链接 → 5 个核心问题" 的量化总结，学习者获得客观的完成感
2. **社会展示**——卡片设计为社交平台适配的比例（1:1、3:1.8），语言自适应的 slogan（"每一次阅读，都不止于阅读。" / "Every read builds something that lasts."），QR 码直连 GitHub——这些不是视觉细节，是降低分享摩擦的认知设计

两道分享卡风格（A: 知识图谱背景，B: 概念图 + 竖排数据）进一步降低了决策成本——随机选择消除了"选哪个好"的认知负担，同时保持每次分享的新鲜感。

#### Cognitive Flow：并行调度的等待设计

并行 agent 调度中 ceil(n/10) + 5 分钟超时降级的改进，直接的认知影响是**减少等待焦虑**。Csikszentmihalyi 的 Flow 理论指出，学习中的断点会打破沉浸状态。之前 ceil(n/5) 导致大量并行 sub-agent 超时，用户必须中断注意力、手动干预。新的方案让 Agent 自主降级，用户无需从学习状态切换到"调试工具"状态。

### 1.3 认知设计系统性评价

v3.1.0 的学习模型仍然是持续建构循环，但加入了一条新的认知路径——

```
Phase 0: 知识状态感知 → Phase 1-5: 输入→编码→整合→提取 → Phase 6: 元认知拓展
                                                                      ↓
                                                              Step 6.5: 社会性学习动机 (分享)
```

从认知加工的单人循环扩展为包含社会分享的双人循环——这是 Vygotsky **社会建构主义**在 Agent 工具设计中的体现。知识不仅是个人的，也是可以展示、讨论和获得反馈的。

### 2.2 关键工程决策

#### 统计规则的工程学意义

v3.1.0 最不明显的改进是统计规则的重新定义，但它是最具工程学意义的——因为它消除了"数据源不可靠"带来的系统性误差：

| 统计项 | 旧方法 | 新方法 | 误差根因 |
|--------|--------|--------|----------|
| 源文件字数 | 从生成笔记反推 | Phase 1.2 直接统计 | 生成笔记是压缩输出，反推有衰减 |
| 原子笔记数 | 从 report 读取 | 扫描 H3 文件夹 + 排除 MOC/路线图 | 之前的统计含 MOC |
| 双链数 | Stage 1 结构亲和（69） | 实际 `[[wikilink]]` 扫描（18） | 结构亲和是所有可能对，不是实际链接 |

这个教训——数据必须来自源头，不能二次推导——已写入 SKILL.md 的验证清单。每次计数错误都是一条工程的因果链。

#### 并行调度设计的迭代轨迹

并行 agent 调度经历了精益的迭代——

1. ceil(n/5) → 大量 sub-agent 超时 → 用户手动干预
2. ceil(n/10) + 5 分钟超时 → 自动降级串行
3. 最终方案：合理负载 + 宽限超时 + 自主降级

这个迭代轨迹体现了"过度工程化"的修正——探测多 agent 并行能力、分批调度都是正确的想法，但在用户真实环境中都是不必要的复杂度。最终收敛到最简单方案：**试并行，卡了就串行**。

#### 静态资产嵌入的架构意义

分享卡从动态 SVG 转换改为 static PNG 嵌入，不仅仅是性能优化（消除 SVG→PNG 转换步骤）。它是一种资产管理的设计模式——**将运行时计算转变为编译时嵌入**。两张模板（share-card.html，share-card-b.html）都内嵌了各自的 base64 背景图，不再需要：

- 运行时 SVG 渲染
- 临时文件路径管理
- 跨风格的文件名冲突处理

模板自包含、零外部依赖、在 `/tmp` 渲染也不会有路径问题——这是 Agent Skill 设计的重要一课。

### 2.3 Phase 6 Deep Research 的 MCP 集成

Phase 6 的 Firecrawl + Exa 集成不是"添加工具"，而是将 VaultForge 的知识检索能力从内置 websearch 的平面搜索升级为**结构化多源研究**。这消除了之前的信息衰减问题（用路线图替代源材料填充笔记），并且通过 MCP 本地配置做到了 API Key 级别控制。

需要注意的是 MCP 工具只在新建会话时注入——续接旧会话不会添加。这个限制已写入 README 的 Prerequisites，用户不会为此困惑。

## 五、核心思考

### 1. Agent Skill 的"编译时 vs 运行时"设计边界

v3.1.0 的分享卡模板演变揭示了一个 Agent Skill 设计的重要模式——**将运行时计算尽可能推向编译时**。graph-bg.svg 的动态渲染曾经是 Step 6.5 中最耗时的步骤。将其预渲染为 PNG 并 base64 嵌入模板，将 10+ 秒的 SVG 渲染压缩到 0，同时消除了临时文件路径冲突。这是 Agent Skill 设计的关键经验：**Agent Skill 不是 web 应用——不需要也不应该做运行时资源转换。**

### 2. 统计数据可靠性的教育意义

源文件字数 → 生成笔记字数 → MOC 数量 → 双链数量这些数字的统计偏差，本质上是信息衰减的另一个表现形式——数据在管道中传递时每次变换都可能引入误差。VaultForge v3.1.0 的经验是：**统计数据只应有一个权威来源，这个来源必须是原始产出的直接读取，而非中间步骤的推算。** 这对任何需要向用户展示量化成果的 Agent 工具都是通用原则。

### 3. 复杂度控制：从"探测与分批"到"直接试，卡了降级"

并行 dispatch 的迭代历程是对过度工程化的自我修正——从 ceil(n/5) 的超时失灵，到添加并行能力探测（增加复杂度），到分批调度（再增加复杂度），最终回到最简单的 ceil(n/10) + 宽限超时 + 自动降级。这个轨迹说明：**Agent Skill 的鲁棒性不能靠增加分支逻辑——要在最简单的逻辑中发现最可靠的收敛点。**
